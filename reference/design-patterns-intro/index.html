Design What...?!

Design patterns are a collection of common solutions to solve common software development design problems

More Info

There are 23 cataloged design patterns.
Even more that are not cataloged or documented.

These 23 standard patterns are classified into 3 categories.

  • Creational Patterns

  • Behavioral Patterns

  • Structural Patterns

Creational Patterns

Creational patterns handle how objects are created. The purpose of creational patterns is to isolate a system from having any knowledge of how objects are created.

  • Singleton

  • Factory Method

  • Abstract Factory

  • Prototype

  • Builder

Behavioral Patterns

Behavioral patterns handle how objects interact. That interaction involves communication with one another, dependencies on one another, isolation from each other to be flexible, maintainable and testable.

  • Observer

  • Iterator

  • Chain of Responsibility

  • Command

  • Strategy

  • Mediator

  • Interpreter

  • State

  • Memento

  • Visitor

  • Template Method

Structural Patterns

Structural Design Patterns are Design Patterns that ease the design by identifying a simple way to realize relationships between entities.

  • Decorator

  • Adapter

  • Facade

  • Bridge

  • Flyweight

  • Proxy

  • Composite

Design Patterns Catalog

Creational Patterns

Abstract Factory
Provide an interface to create families of related objects without specifying their concrete classes
Builder
Seperate the construction of a complex object from its representation so that the same construction process can create different representations.
Factory Method
Define an interface for creating an object, but let subclasses decide which class to instantiate.
Prototype
Specify the kind of objects to create using a prototypical instance and create new objects by copying this prototype.
Singleton
Ensure a class has only one instance and provide a global point of access to it.

Behavioral Patterns

Chain of Responsibility
Chain the recieving objects and pass the request along the chain until an object handles it.
Command
Encapsulate a request as an object allowing clients with different requests to be passed as parameters.
Interpreter
Given a language defined by a grammar along with an interpreter that uses the grammar to interpret the language
Iterator
Provide an interface to access the elements of an aggregate object sequentially, without exposing the object's implementation.
Mediator
Define an Object that encapsulates a set of object thus keeping them from referring to each other explicitly and vary the interaction independently.
Memento
Capture an object's internal state without violating encapsulation, so that the object can be restored to the state later.
Observer
Define a one-to-many relationship between objects, so that when one object changes, it notifies to all it's dependencies.
State
Allow an object to change it's behaviour when it's internal state changes.
Strategy
Create a family of algorithms and make them interchangable by encapsulating them.
Template Method
Define the steps of an algorithm/operation and allow subclasses define each step, but preserving the order of the steps.
Visitor
Represent an operation to be performed, so that new operations can be defined without changing the elements operated on.

Structural Patterns

Adapter
Provide a way to convert the interface of a class to a different interface required by the client.
Bridge
Decouple an abstraction from it's interface allowing the two to vary independently.
Composite
Compose a set of objects in an heirarchy allowing clients to treat individual objects and collection of objects equally.
Decorator
Adding additional functionality to an object dynamically, thus making any object extendable.
Facade
Define a unified interface for a set of related interfaces, making them easy to use.
Flyweight
Create light weight objects in large number by sharing data among the objects.
Proxy
Provide a placeholder for another object to control access to it.
